# 오타 찾기 및 수정

이 skill은 한국어(또는 한국어+영어 기술 용어 혼합) 텍스트에서 오타를 찾아 수정한다.

## 핵심 원칙

**오타만 고친다. 다른 것은 절대 건드리지 않는다.**

- 의미를 바꾸지 않는다
- 문체를 바꾸지 않는다
- 영어 기술 용어를 한국어로 "번역"하지 않는다
- 고유 명사(회사명, 제품명, 프레임워크명)는 손대지 않는다
- 코드 블록 내부는 손대지 않는다

## 무엇이 "오타"인가

### 확실한 오타 (자신 있게 수정)

- **자판 오기**: "코딘" → "코딩", "에이정트" → "에이전트"
- **받침 누락/추가**: "박혀잇다" → "박혀있다", "햇다" → "했다"
- **명백한 띄어쓰기 오류**: "할 수있다" → "할 수 있다", "그것은이다" → "그것은 ~이다"
- **흔한 맞춤법 오류**: "되" vs "돼" (과거형 "됬다" → "됐다"), "왠" vs "웬" ("웬지" → "왠지"), "안" vs "않"

### 의심되는 부분 (사용자 확인 후 수정)

- 외래어 표기 흔들림: "콘텐츠" vs "컨텐츠" — 표준은 "콘텐츠"지만 프로젝트 내 일관성 우선
- 합성어 띄어쓰기: 사전마다 다른 경우 (예: "데이터베이스" vs "데이터 베이스")
- 한글-영어 경계 띄어쓰기: "AI 시대" vs "AI시대"

### 절대 건드리지 않는 것

- **영어 기술 용어**: `context window`, `subagent`, `harness`, `MCP`, `Claude Code`, `worktree`, `prompt`, `token` 등 — 이건 오타가 아니라 의도된 영어 표기다
- **고유 명사**: Anthropic, GitHub, Reveal.js, Pretendard, JetBrains, Mermaid 등
- **의도적 비격식체**: 슬라이드 발표용 "~한다", "~다" 종결 — 이걸 "~합니다"로 바꾸면 안 된다
- **인용문 원문**: `<blockquote>` 안의 영어 인용은 그대로 둔다
- **코드 블록**: `<code>`, `<pre>`, ``` ``` ``` ``` 안 — 코드는 그렇게 써져 있을 이유가 있다
- **HTML 속성값**: `class="..."`, `id="..."` 안의 영어
- **frontmatter**: YAML `---` 영역

## 처리 절차

1. 대상 파일 경로 확인. 컨텍스트에서 추론, 없으면 사용자에게 묻기.
2. 파일을 Read한다.
3. **텍스트 노드만** 스캔한다. 다음은 건너뛴다:
   - HTML 속성 (`class=`, `id=`, `style=`, `data-*`)
   - 코드 블록 (`<code>...</code>`, `<pre>...</pre>`, ``` ```)
   - frontmatter (`---` 사이)
   - URL / 파일 경로
4. 발견한 후보를 3개 카테고리로 분류한다.
5. 결과를 사용자에게 표 형태로 제시한다 — 수정 전에 반드시.
6. 사용자 확인을 받은 후 Edit tool로 일괄 수정.
7. 수정 후 변경된 라인을 다시 보여준다.

## 결과 보고 형식 (예시)

```
파일: slides/02-11-why-multi-agent.md

확실한 오타 (3개):
  L7  "병렬 추로"     → "병렬 추론"
  L23 "후보 전화 2배" → "후보 전환 2배"
  L37 "납득해야함"    → "납득해야 함"

의심 (1개) — 사용자 확인 필요:
  L19 "컨텐츠"        → "콘텐츠" (표준 표기, 단 프로젝트 일관성 확인 필요)

건드리지 않음 (2개):
  L5  "Multi-agent systems" — 영어 인용문
  L34 "Anthropic"           — 고유 명사

수정하시겠습니까? (확실한 오타 3개만 / 의심 포함 4개 / 직접 선택)
```

## 좋은 예

원문 (가상):
```html
<section>
  <h2>바이브 코딘이란 무엇인가</h2>
  <p>요즘 시니어 개발자들이 <code>context window</code>를 어떻게 다룰지 고민하고 잇다.</p>
  <p>특히 Anthropic 보고서가 나온 후로 더 그렇다.</p>
  <pre><code>const x = 1;  // 오타 처럼 보이지만 코드</code></pre>
</section>
```

스캔 결과:
- L2: "코딘" → "코딩" (확실한 오타)
- L3: "잇다" → "있다" (확실한 오타)
- L3: `context window` — 무시 (영어 기술 용어, 그것도 `<code>` 안)
- L4: "Anthropic" — 무시 (고유 명사)
- L5: 코드 블록 — 무시 (내부 일절 손대지 않음)

수정 결과:
```html
<section>
  <h2>바이브 코딩이란 무엇인가</h2>
  <p>요즘 시니어 개발자들이 <code>context window</code>를 어떻게 다룰지 고민하고 있다.</p>
  ...
```

## 잘못된 행동 (Red Flags — 즉시 멈출 것)

| 잘못된 행동 | 왜 잘못인가 |
|---|---|
| `subagent` → `서브에이전트`로 변경 | 영어 기술 용어를 임의로 번역. 의도된 표기다 |
| `"~한다"` → `"~합니다"` 변경 | 슬라이드 비격식체는 의도된 톤. 문체 변경은 오타 수정이 아니다 |
| 사용자 확인 없이 의심 항목 수정 | "확실"과 "의심"의 분리는 핵심 — 의심은 항상 확인 |
| 코드 블록 내부 수정 | 코드는 의도된 형태. 오타로 보여도 그대로 둔다 |
| diff/표 없이 일괄 수정 | 사용자가 무엇이 바뀌었는지 알 수 없음 |
| 한 번에 여러 파일을 동시 수정 | 한 파일씩, 사용자 확인 후 진행 |
